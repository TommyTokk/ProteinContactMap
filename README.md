## Overview

This repository contains the project developed for the **Computing and Networking: Resources and Tools** module of the Master's Degree in **Informatics for Digital Health** at the University of Pisa.

The project studies the performance of different execution strategies for computing **protein contact maps**. In particular, it compares:

- a **sequential implementation**;
- an **OpenMP implementation**, varying the number of threads;
- a **hybrid MPI + OpenMP implementation**, varying both the number of processes and the number of threads.

The goal is to evaluate how different parallelisation strategies affect execution time, speedup, efficiency, and computational cost.

## Problem Description

A **protein contact map** is a simplified two-dimensional binary representation of a protein's three-dimensional structure. It preserves relevant spatial relationships between residues while discarding absolute Cartesian coordinates. This reduces dimensionality and removes rotational and translational variance, making contact maps useful in computational workflows such as structure prediction, fold recognition, and protein design.

For a protein with `N` amino acid residues, the contact map is represented as an `N × N` symmetric binary matrix `C`. Each element `C_ij` indicates whether residues `i` and `j` are within a predefined distance threshold:

```math
C_{ij} =
\begin{cases}
1 & \text{if } d_{ij} \le d_c \\
0 & \text{otherwise}
\end{cases}
```

where:

- $d_{ij}$ is the spatial distance between residues `i` and `j`;
- $d_c$ is the contact threshold distance.

Typical values for $d_c$ range from `8 Å` to `12 Å`, depending on the reference atoms and the type of interaction being studied, such as hydrogen bonds, hydrophobic packing, or salt bridges.

### Distance Computation

In this project, residue distances are computed using the `Cα` atoms. Given the coordinates of two `Cα` atoms:

```math
(x_i, y_i, z_i) \quad \text{and} \quad (x_j, y_j, z_j)
```

the Euclidean distance is:

```math
d_{ij} = \sqrt{(x_i - x_j)^2 + (y_i - y_j)^2 + (z_i - z_j)^2}
```

To avoid the cost of computing the square root, the implementation compares squared distances:

```math
d_{ij}^2 = (x_i - x_j)^2 + (y_i - y_j)^2 + (z_i - z_j)^2
```

Therefore, the contact map entry is computed as:

```math
C_{ij} =
\begin{cases}
1 & \text{if } d_{ij}^2 \le d_c^2 \\
0 & \text{otherwise}
\end{cases}
```

## Experimental Design

The project evaluates three experimental configurations:

1. **Optimised execution**  
   Sequential, OpenMP, and hybrid MPI + OpenMP implementations are compiled and executed with all intended optimisations enabled.

2. **Execution without sequential optimisations**  
   The sequential implementation is intentionally compiled and executed without selected optimisations, while the OpenMP and hybrid implementations remain optimised.

3. **Execution with workload injection**  
   Artificial extra floating-point operations are added to the distance computation to increase the computational cost per iteration and evaluate how this affects parallel scalability.

---

## Experiment 1: Optimised Execution

In the optimised version, the implementation uses a **Structure of Arrays** layout instead of an **Array of Structures** layout. This improves memory access patterns because coordinates are stored in contiguous arrays.

The `Model` structure is defined as follows:

```cpp
typedef struct Model {
    std::vector<float> X;
    std::vector<float> Y;
    std::vector<float> Z;

    void resize(int n) {
        X.resize(n);
        Y.resize(n);
        Z.resize(n);
    }
} Model;
```

### Sequential Implementation

The sequential implementation uses the following function in `seq_main.cpp`:

```cpp
std::vector<uint8_t> get_residue_distances_soaV2(const Model& model, int size)
```

This function uses the Structure of Arrays layout and removes explicit `if/else` branches from the inner computation. The code is also compiled with the `-O3` flag, enabling compiler optimisations such as vectorisation and loop transformations.

### OpenMP Implementation

The OpenMP implementation uses the following function in `omp_main.cpp`:

```cpp
std::vector<uint8_t> get_residue_distances_omp_soa(
    const Model& model,
    int size,
    int n_threads
)
```

This version uses the same memory layout and branch-removal strategy as the optimised sequential implementation. The outer loop is parallelised with OpenMP using **static scheduling**. Since the number of chunks is not explicitly specified, OpenMP divides the iteration space into approximately equal contiguous chunks, with at most one chunk assigned to each thread.

The inner loop also uses a `simd` directive to encourage SIMD vectorisation. This may partially overlap with the automatic vectorisation enabled by the `-O3` compiler flag.

### Hybrid MPI + OpenMP Implementation

The hybrid implementation combines MPI and OpenMP. The MPI master process, `rank 0`, partitions the matrix by rows and distributes the work among the available processes. The master process also participates in the computation. At the end of the computation, the partial contact maps are gathered using `MPI_Gatherv`.

Given an `N × N` matrix and `P` processes, each process computes approximately an `(N / P) × N` submatrix.

The function used to compute each submatrix is:

```cpp
std::vector<uint8_t> get_residue_distances_mpi_soa(
    const Model& model,
    int size,
    size_t starting_row,
    size_t count,
    int n_threads
)
```

OpenMP threads are then used inside each MPI process to parallelise the local computation.

### Running the Optimised Experiment

> [!WARNING]
> Before running this experiment, make sure that each main file calls the correct optimised function and that the `-O3` flag is enabled in the `CXXFLAGS` line of the `makefile`.

Compile all implementations:

```bash

```

This creates the executables inside the `bin/` directory.

To run the full benchmark suite:

```bash

```

The benchmarking script supports the following options:

```text

--seq  Run only the sequential benchmark
--omp  Run only the OpenMP benchmark
--mpi  Run only the MPI benchmark
```

Unless configured otherwise, benchmark outputs are saved in `benchmark_results/`.

To compute statistics such as speedup, efficiency, and cost, and to generate plots:

```bash

```

## Experiment 2: Execution Without Sequential Optimisations

In this experiment, the sequential implementation is intentionally modified to remove selected optimisations.

Instead of using the Structure of Arrays layout, the code uses the `Atom` structure:

```cpp
typedef struct Atom {
    char record_name[7];                 // Contains ATOM or HETATM
    int serial_number;                   // Atom serial number
    char name[5];                        // Atom name
    char alternate_locator_indicator[2];
    char residue_name[4];                // Residue name
    char chain_id[2];                    // Chain identifier
    int res_seq;                         // Residue serial number
    char code_residue_insert[2];         // Residue insertion code
    float x, y, z;                       // Atom coordinates
    float occupancy;                     // Occupancy
    float temp;                          // Temperature factor
    char segment_id[5];                  // Segment identifier
    char element_sym[3];                 // Element symbol
    char charge[3];                      // Charge
    int model;
} Atom;
```

With this layout, atomic coordinates are not stored contiguously in memory, increasing memory-access overhead. The implementation also uses an explicit `if` statement, which introduces branch-evaluation overhead. In addition, the sequential code is compiled without the `-O3` flag, so compiler optimisations such as vectorisation and loop unrolling are not applied.

### Sequential Implementation

For this experiment, only the sequential implementation is changed. The function called from `seq_main.cpp` is:

```cpp
std::vector<uint8_t> dm = get_residue_distances(alphas_vec);
```

This version is intentionally implemented without the optimisations used in Experiment 1.

### Running the Non-Optimised Sequential Experiment

> [!WARNING]
> Before running this experiment, make sure that `seq_main.cpp` calls the non-optimised function. The sequential code must be compiled **without** `-O3`, while the OpenMP and hybrid MPI + OpenMP implementations must still be compiled **with** `-O3`.

First, compile the OpenMP and MPI implementations with optimisations enabled:

```bash
make omp mpi
```

Then remove the `-O3` flag from the sequential compilation flags. For example, in the `makefile`, comment the optimised `CXXFLAGS` line and uncomment the non-optimised one.

After modifying the `makefile`, compile the sequential implementation:

```bash
make seq
```

> [!NOTE]
> Before compiling the sequential implementation, ensure that `seq_main.cpp` uses the non-optimised protein contact map computation.

Finally, run the benchmark and process the results:

```bash
./scripts/benchmark_all.sh --all
./scripts/results_script.py
```

## Experiment 3: Execution With Workload Injection

In the final experiment, additional floating-point operations are injected into the distance computation to simulate a heavier workload. The following code is added after the squared-distance computation:

```cpp
// Extra operations to increase execution time
for (int z = 0; z < N_ITERATIONS; z++) {
    dist_sq += 0.000001f;
    dist_sq -= 0.000001f;
}
```

This modification is applied to the sequential, OpenMP, and hybrid MPI + OpenMP implementations. The purpose is to increase the computation-to-overhead ratio and evaluate whether the parallel implementations benefit more from a heavier per-iteration workload.

### Sequential Implementation

The sequential implementation uses the following function in `seq_main.cpp`:

```cpp
std::vector<uint8_t> get_residue_distances_seq_inj(
    const Model& model,
    int size
)
```

This function follows the same logic as the optimised Structure of Arrays implementation, but adds the extra workload after computing each squared distance.

### OpenMP Implementation

The OpenMP implementation uses the following function in `omp_main.cpp`:

```cpp
std::vector<uint8_t> get_residue_distances_omp_inj(
    const Model& model,
    int size,
    int n_threads
)
```

This version is equivalent to the optimised OpenMP implementation, with the additional injected workload.

### Hybrid MPI + OpenMP Implementation

The hybrid implementation uses the following function in `mpi_main.cpp`:

```cpp
std::vector<uint8_t> get_residue_distances_mpi_inj(
    const Model& model,
    int size,
    size_t starting_row,
    size_t count,
    int n_threads
)
```

This version follows the same logic as the optimised hybrid implementation, with the additional floating-point operations included in the distance computation.

### Running the Workload-Injection Experiment

> [!WARNING]
> Before running this experiment, make sure that each main file calls the workload-injection function and that the `-O3` flag is enabled in the `CXXFLAGS` line of the `makefile`.

Compile all implementations:

```bash

```

Run the full benchmark suite and process the results:


## Metrics Computed for Each Experiment

For each experimental configuration, 50 independent runs are performed to ensure statistical stability. Let $i \in \{1, \dots, 50\}$ denote the run index.

The representative execution times are defined as the minimum observed values over the 50 runs.

**Sequential execution time:**

$$
T_{	\text{seq}} = \min_i \left( t_{\text{seq}}^{(i)} 
\right)
$$

**OpenMP execution time with `t` threads:**

$$
T_{\text{omp}}(t) = \min_i \left( t_{\text{omp}}^{(i)}(t) 
\right), \quad \forall t \in \mathcal{T}
$$

**Hybrid MPI + OpenMP execution time with `p` MPI processes and `t` OpenMP threads:**

$$
T_{\text{mpi}}(t,p) = \min_i \left( t_{\text{mpi}}^{(i)}(t,p) 
\right), \quad \forall t \in \mathcal{T}, \forall p \in \mathcal{P}
$$

where $\mathcal{T}=\{4, 8, 16, 32, 64, 128, 255\}$ and $\mathcal{P} = \{4, 8, 16, 32, 64\}$ represent the sets of thread counts and MPI process counts tested, respectively.

For each parallelisation technique, three metrics are computed: **speedup**, **efficiency**, and **cost**.

### OpenMP Metrics

For the OpenMP implementation, the metrics are defined as follows.

**Speedup:**

$$
SP_{\text{omp}}(t) = \frac{T_{\text{seq}}}{T_{\text{omp}}(t)}, \quad t \in \mathcal{T}
$$

**Cost:**

$$
C_{\text{omp}}(t) = T_{\text{omp}}(t) 	\times t, \quad t \in \mathcal{T}
$$

**Efficiency:**

$$
E_{\text{omp}}(t) = \frac{SP_{\text{omp}}(t)}{t} = \frac{T_{\text{seq}}}{T_{\text{omp}}(t) \times t}, \quad t \in \mathcal{T}
$$

In the OpenMP case, speedup measures how much faster the parallel implementation is compared with the sequential baseline. Cost is the execution time multiplied by the number of threads. Efficiency measures how effectively the available threads are used.

### Hybrid MPI + OpenMP Metrics

For the hybrid MPI + OpenMP implementation, the metrics are defined as follows.

**Speedup:**

$$
SP_{\text{mpi}}(t,p) = \frac{T_{\text{seq}}}{T_{\text{mpi}}(t,p)}, \quad t \in \mathcal{T}, \quad p \in \mathcal{P}
$$

**Cost:**

$$
C_{\text{mpi}}(t,p) = T_{\text{mpi}}(t,p) \times t \times p, \quad t \in \mathcal{T}, \quad p \in \mathcal{P}
$$

**Efficiency:**

$$
E_{\text{mpi}}(t,p) = \frac{SP_{\text{mpi}}(t,p)}{t \times p} = \frac{T_{\text{seq}}}{T_{\text{mpi}}(t,p) \times t \times p}, \quad t \in \mathcal{T}, \quad p \in \mathcal{P}
$$

In the hybrid case, the total number of parallel workers is considered to be $t 	\times p$, because each MPI process uses $t$ OpenMP threads. Therefore, efficiency and cost account for both levels of parallelism.
